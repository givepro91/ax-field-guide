# AX Field Guide

백엔드 엔지니어로 시작해 프로덕트 리드 일을 하면서, AX(AI Transformation)를 현장에서 배우는 동안 정리하는 작업 노트다. 완성된 방법론을 전달하려는 글이 아니라, 무엇을 해봤고 어떤 기준으로 판단했는지를 그때그때 남기는 기록에 가깝다.

## 왜 이걸 쓰나

요즘 AX라는 말이 너무 넓게 쓰인다. 챗봇을 하나 붙여도 AX, 회의록을 자동 요약해도 AX다. 그 말들이 틀린 건 아니지만, 그 수준에서 멈추면 "AI를 한번 써봤다"와 "일하는 방식을 바꿨다"가 구분되지 않는다.

내가 보기에 둘은 다른 일이다. 그래서 내가 쓰는 정의부터 분명히 해두려 한다.

> AX는 AI를 도입하는 일이 아니라, 사람이 반복적으로 판단하던 업무 구조를 AI·데이터·자동화·승인 흐름으로 다시 설계하는 일이다.

이 정의가 맞는지, 실무에서 어떻게 작동하는지는 한 번에 알 수 없다. 도구를 써보고, 자동화를 만들었다 접어보고, 승인 흐름을 다시 그려보면서 확인하는 수밖에 없다. 이 모음은 그 확인 과정을 공개적으로 남기는 자리다.

## 백엔드·프로덕트 경험과 AX

AX를 처음부터 새로 배우는 분야로 느끼지는 않는다. 지금까지 하던 일의 연장선에 가깝다.

백엔드를 하면서 몸에 밴 건 시스템이 **언젠가 실패한다**는 전제다. 재시도, 멱등성, 롤백, 관측, 알림 — 정상 동작보다 비정상 상황을 먼저 설계한다. AI 자동화도 결국 한 노드가 비결정적으로 움직이는 분산 시스템이고, 같은 질문이 그대로 적용된다. 틀렸을 때 어떻게 되돌리나, 무슨 일이 있었는지 어떻게 추적하나.

프로덕트 리드를 하면서 배운 건 기술보다 **문제 선택**이 먼저라는 점이다. 누가 어떤 결정을 내리려 하고, 진짜 병목이 어디인지를 잘못 잡으면 좋은 구현도 쓸모가 없다. AX에서 가장 흔한 실패는 모델이 부족해서가 아니라, 바꿀 필요가 없는 일을 바꾸거나 바꿔야 할 일을 못 고르는 데서 온다.

이 두 경험이 만나는 지점에 AX가 있다고 본다. 그래서 이 노트는 모델 성능표나 도구 자랑이 아니라, **문제를 어떻게 구조화했고 어디에 무엇을 붙였으며 무엇을 포기했는지**를 중심에 둔다.

## 여기서 다루는 것

- AX를 내 언어로 정의하고, 그 기준이 실무에서 어떻게 쓰이는지
- AX를 맡은 사람이 실제로 하는 일 — 문제 선택, 도구 선택, 승인 설계, 운영
- Claude Code·Codex·Cursor·MCP·에이전트 워크플로우를 직접 써본 필드 리뷰
- 승인·롤백·감사·품질 게이트 같은 반복 설계 패턴
- 만들었다가 접은 자동화와 거기서 배운 것

다루지 않는 것은 AI 뉴스 요약, 벤치마크 경쟁, 단발 프롬프트 팁, 그리고 식별 가능한 회사 내부 사례다. 사례는 모두 익명화해서 패턴만 남긴다.

## 읽는 사람이 기대할 수 있는 것

확실한 답보다는 **판단의 근거**다. "이 도구가 최고다" 같은 결론이 아니라, 어떤 상황에서 무엇을 골랐고 왜 그랬는지, 다시 한다면 무엇을 바꿀지를 적는다. 일부 글은 결론이 아직 안 난 채로 남아 있을 수 있다. 그건 의도된 것이다 — 현장에서 배우는 중이라는 뜻이고, 결론이 분명해지면 갱신한다.
