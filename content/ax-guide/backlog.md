# Backlog — 앞으로 쓸 글

다음에 쓸 글 목록이다. 순서는 고정이 아니라, 실제로 써보고 정리할 거리가 생긴 것부터 쓴다. 각 항목에는 그 글에서 답하려는 질문 한 줄을 같이 적어둔다 — 질문이 분명하지 않으면 아직 쓸 때가 아니다.

상태 표기: `초안 준비` 바로 쓸 수 있음 · `재료 모으는 중` 경험이 더 필요함 · `용어 확인 필요` 쓰기 전 사실 확인 필요.

## 1. 정의·관점

- **AX Engineer / AX Lead는 무엇을 해야 하는가** — `초안 준비`
  하루를 무엇으로 채우는가. 모델을 고르는 사람인가, 문제를 고르는 사람인가, 승인 흐름을 그리는 사람인가. 역할의 경계와 안 하는 일.
- **어떤 업무 루프를 먼저 AX 대상으로 고를 것인가** — `초안 준비`
  빈도·경계·회복 가능성으로 후보를 거르는 기준. 시연용 루프와 가치 있는 루프가 다른 이유. (`what-is-ax`의 1번을 한 편으로 확장.)
- **AI 출력의 신뢰도를 어떻게 따질 것인가** — `재료 모으는 중`
  "믿어도 되나"를 감이 아니라 기준으로 답하기. 정확도, 회복 비용, 검증 비용의 관계.

## 2. 도구 필드 리뷰

AX 관점 = 역할·한계·언제 쓰나. 성능 비교가 아니라 어떤 일에 무엇을 붙였더니 어땠는지.

- **Claude Code를 AX 관점에서 쓰는 법** — `재료 모으는 중`
  코딩 도구를 넘어 반복 작업을 어디까지 위임할 수 있나. 승인·검증을 어디에 두나.
- **Codex를 AX 관점에서 쓰는 법** — `재료 모으는 중`
  같은 질문을 Codex에 대해. 강점이 갈리는 지점.
- **Claude Code vs Codex** — `재료 모으는 중`
  둘 중 하나를 고르는 글이 아니라, 어떤 작업 성격에 무엇이 맞는지. 위 두 글이 쌓인 뒤에 쓴다.
- **Cursor / Claude Code / Codex의 역할 분리** — `재료 모으는 중`
  하나로 다 하려 하지 않기. IDE 보조·터미널 에이전트·작업 위임의 경계.
- **MCP는 AX에서 어떤 의미가 있는가** — `재료 모으는 중`
  도구·데이터를 모델에 연결하는 표준이라는 점이 AX 설계(특히 데이터 접근·워크플로우 연결)에서 갖는 의미. 한계와 운영 부담도.

## 3. 패턴·플레이북

`what-is-ax`의 4·5번을 구체적인 설계 패턴으로 푼다.

- **AI agent에 승인 구조가 필요한 이유** — `초안 준비`
  자동 적용 vs 제안-후-승인. 위험 크기에 따라 게이트를 어디에 두는가.
- **Human-in-the-loop 패턴** — `재료 모으는 중`
  사람이 "아니오"를 말할 수 있는 자리를 어디에, 몇 군데 둘 것인가. 개입이 형식이 되지 않게 하는 법.
- **AI 자동화의 audit trail 설계** — `재료 모으는 중`
  무슨 입력→무슨 출력→누가 승인을 어떻게 남기나. 나중에 추적 가능하게 만드는 최소 구성.
- **운영 자동화는 언제 폐기해야 하는가** — `초안 준비`
  만드는 것보다 접는 판단이 어렵다. 드리프트·비용·신뢰 하락의 신호와 폐기 기준. (안티패턴 축의 첫 글이 될 수 있음.)

## 4. 개념 정리

도구 이전의 개념들. 헷갈리는 용어를 내 기준으로 구분한다.

- **RAG와 Agent Workflow의 차이** — `재료 모으는 중`
  검색으로 맥락을 채우는 것과, 여러 단계를 스스로 밟게 하는 것. 언제 무엇으로 충분한가.
- **Loop Engineering이란 무엇인가** — `용어 확인 필요`
  에이전트를 반복 루프로 돌려 작업을 끝까지 끌고 가는 접근으로 이해하고 있으나, 통용되는 정의·범위가 정리된 용어인지 확인 후 내 기준으로 정의할 것.
- **"랄프 / RALF" 관련 개념** — `용어 확인 필요`
  에이전트를 반복 루프로 돌리는 코딩 기법으로 들었으나 **단정하지 않는다.** 쓰기 전에 확인할 것: (1) 정확한 명칭이 "Ralph"인지 "RALF"인지, 약어라면 무엇의 약자인지, (2) 출처와 최초 제안자, (3) `Loop Engineering`과 같은 것인지 다른 것인지. 사실 확인 전에는 정의를 적지 않는다.

## 글 고르는 기준 메모

- 결론이 안 났어도 질문이 분명하면 쓴다. 다만 "확인 필요" 항목은 사실 확인 전에 발행하지 않는다.
- 도구 리뷰는 직접 써본 뒤에만 쓴다. 안 써보고 비교하는 글은 이 프로젝트가 피하려는 바로 그 글이다.
- 한 편이라도 회사 내부가 특정되면 익명화하거나 보류한다. (기준: [strategy/ax-field-guide.md](../../strategy/ax-field-guide.md)의 "공개 위험 기준".)
